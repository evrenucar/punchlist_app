The one-paragraph version

The product is one HTML file. Three source files (src/) get inlined by a 30-line build script into outputs/task-board.html, which runs from a double-click with zero dependencies, zero network requests, and no server. Everything else in the repo exists to build it, test it, host it, or coordinate the humans and agents working on it.

Dependencies across the whole repo: zero. No package.json, no node_modules, no framework. That's a deliberate rule, not an accident.

The three layers

  • The app (src/ → outputs/, website/): the product. Your board on the laptop and phone.
  • The development interface (status/): this page's sibling at localhost:4173. It embeds the real built app in an iframe seeded from status-board.json, and adds chat, the staircase graph, agent presence, and the braindump flow. It never copies app code: the whole coupling is three DOM hooks and a small API the app exposes, guarded by a test.
  • The website (website/): the landing page, the build notes, and a fresh copy of the app. Pushing main deploys it to GitHub Pages.

Where your data lives

your board   → localStorage  key scheduling-task-management-board-v1  (text, ~KB now)
your images  → IndexedDB     punchlist-assets-v1                      (bytes, since v1.4.0)
sync         → private GitHub repo: punchlist-board.json + assets/<id>.webp, one file per image
device name  → its own localStorage key, never syncs
sync token   → its own localStorage key, never lands in any export or sync payload
signing key  → inside board state, syncs to YOUR devices only, stripped from every export

Since v1.4.0 (July 19) the board file itself carries no image bytes: each image is an immutable record in IndexedDB plus one file in the sync repo's assets/ folder. Push uploads what the remote is missing before the board that references it; pull downloads what your device is missing; nothing transfers twice. Exports re-embed everything, so a backup file is still complete on its own.

Rules the code holds itself to

  • Task IDs are immutable and never encode position. Aliases, shortcuts, trash restore, and scheduling all depend on that.
  • Where a task IS and where it was CREATED are different facts; moving never rewrites creation metadata.
  • Exports strip settings, identity, contacts. A test fails the build if the private key ever leaks into an export.
  • Disabled features render zero UI. Timeline, reminders, and policy overrides are invisible until switched on.
  • Renders are scoped since July 19: editing swaps one task's subtree in the DOM (8–21ms) instead of rebuilding the board (75–150ms). Full renders remain only for operations that fan out, like linked copies.
  • Every behavior change ships with a regression test, a full suite run, a rebuild, and a real-browser check at phone and desktop widths.

Recent structural changes (July 19)

  • Assets v1.4.0: image bytes out of the board, described above. Retires the 1MB sync read cap and the ~5MB localStorage ceiling.
  • Touch gesture hardening: a press wipes stale gesture state (the intermittent dead-toggles ghost), native OS drag is suppressed on touch, auto-scroll got a gentle ramp with an edge clamp.
  • The wrapper stopped reloading: agent board edits now apply inside the running board through applyExternalState; selection and scroll survive.
